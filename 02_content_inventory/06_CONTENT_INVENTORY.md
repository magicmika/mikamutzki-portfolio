## Purpose

This file tracks known, missing, and potential content and assets
for the portfolio, across Phase 1 and later phases.

It answers:

- What content and assets do we actually have?
- What is missing?
- What is confirmed?
- What is only an idea?
- What may be used for Phase 1?

This file is not written from memory. Every row must be filled in
and confirmed by Mika before it is treated as true. Until confirmed,
an item stays at status `missing` or `needs review`.

This file does not contain design tokens, QA criteria, or case
narratives. It only tracks the existence, status, and location
of content and assets.

---

## Status values

| Status | Meaning |
|---|---|
| `confirmed` | Mika has confirmed this item exists, is correct, and is ready to use |
| `available` | The item exists and is accessible, but has not yet been reviewed or confirmed |
| `needs review` | The item exists in some form but must be checked before it can be trusted or used |
| `missing` | The item does not exist yet and must be created or sourced |
| `not for Phase 1` | The item is relevant to the project but intentionally excluded from Phase 1 |
| `archive only` | The item exists only as historical or old material and is not current truth |

## How to use this file

1. Replace each `TBD` row with a real, named item once Mika identifies it.
2. Never mark an item `confirmed` without Mika explicitly confirming it.
3. Update the `Can be used for Phase 1?` column to `yes` as soon as
   an item reaches `confirmed` status and is scoped for Phase 1.
4. If an item's status changes, update the row directly ‚Äî
   do not duplicate rows.
5. Add new rows as new content or assets are identified.
6. Do not delete rows for content that turns out to be
   `not for Phase 1` or `archive only` ‚Äî keep them visible
   so decisions stay traceable.
7. This file must be reviewed before any Phase 1 page is
   treated as content-complete.
